# 8. Five-way theming: tokens, pre-paint script, AA-adjusted palettes

Date: 2026-07-09

## Status

Accepted

## Context

The site offers system, light, dark, solarized-light and solarized-dark. The
chosen theme must apply before first paint (no flash), follow the OS while
set to "system", persist across visits (ADR 0013), and meet WCAG AA (4.5:1)
for all text — which the classic Solarized palette does not, out of the box.

## Decision

Themes are token sets (`--bg`, `--surface`, `--text`, `--muted`, `--accent`,
`--accent-strong`, `--focus`) on `:root` and `[data-theme=…]` in
`global.css`; components consume tokens through their shadow boundaries. A
tiny inline `<head>` script (generated by `src/client/themeInit.ts`, logic
mirrored from `resolveConcreteTheme`) reads the stored settings and sets
`data-theme` plus `color-scheme` before the first frame. `ThemeController`
keeps a `prefers-color-scheme` listener so "system" follows the OS live.

Every text/background pair was verified numerically against WCAG AA; the
solarized values are adjusted where the classic palette falls short (body
text uses base01 `#586e75` on light, a lightened `#9db1b1` on dark; muted and
accent values tuned to ≥ 4.5:1 on both `--bg` and `--surface`). The amber
accent ties to "photo/light" in the site's name.

## Consequences

Adding a theme is one CSS block plus one entry in `THEMES`. The inline script
and the controller share logic by convention, guarded by a unit test that
evaluates the generated script in a sandbox. Solarized purists will notice
the adjusted values; readers with low vision will not have to.
